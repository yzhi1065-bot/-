## 2. 混合架构详解

### 2.1 架构模式说明

本系统采用 **桌面+云端混合架构**：

| 层次 | 部署位置 | 职责 | 关键技术 |
|------|---------|------|---------|
| **采集端** | 本地诊所桌面 | 连接硬件设备、实时数据采集、本地预处理、离线存储 | Electron/WPF, USB/Bluetooth SDK |
| **业务端** | 云端服务器 | 业务逻辑、数据持久化、用户管理 | 微服务 (Java/Go + Spring/Dubbo) |
| **AI中台** | 云端GPU服务器 | LLM推理、视觉分析、知识图谱推理 | 私有化LLM / API调用 |
| **客户端** | 浏览器/手机 | 医生Web界面、患者移动端 | React/Vue, Flutter/UniApp |

### 2.2 数据流设计

```
采集流程：
┌──────────┐   ┌────────────┐   ┌──────────┐   ┌────────┐
│ 硬件设备   │──▶│ 桌面采集端  │──▶│ 云端API  │──▶│ AI模型 │
│ (脉诊仪等) │   │ (本地缓存)  │   │ Gateway  │   │ 推理   │
└──────────┘   └────────────┘   └──────────┘   └────┬───┘
                                                     │
┌──────────┐   ┌────────────┐   ┌──────────┐        │
│ 患者APP   │──▶│ 云端服务   │──▶│ 数据库   │◀───────┘
│ (预问诊)  │   │            │   │          │
└──────────┘   └────────────┘   └──────────┘
                                                     │
┌──────────┐   ┌────────────┐   ┌──────────┐        │
│ 医生界面  │◀──│ 诊断报告   │◀──│ 业务服务  │◀───────┘
│ (结果展示)│   │ 生成       │   │          │
└──────────┘   └────────────┘   └──────────┘
```

## 3. 技术选型

### 3.1 前端技术栈

| 领域 | 技术选型 | 说明 |
|------|---------|------|
| **桌面采集端** | Electron + Vue 3 / C# WPF | 本地硬件驱动、USB/蓝牙通信 |
| **PC管理端** | React / Vue 3 (Element Plus) | 医生工作站管理界面 |
| **患者端** | 微信小程序 + UniApp | 预问诊、报告查看 |
| **UI框架** | Ant Design / Element Plus | 中后台组件库 |
| **桌面框架** | Tauri (Rust) / Electron | 轻量级桌面容器 |

### 3.2 后端技术栈

| 领域 | 技术选型 | 说明 |
|------|---------|------|
| **主语言** | Go / Java (Spring Boot) | 高并发微服务 |
| **API网关** | Kong / Nginx + Lua | 认证限流 |
| **服务注册** | Nacos / Consul | 服务发现与配置 |
| **消息队列** | RabbitMQ / Kafka | 异步任务与事件驱动 |
| **容器编排** | Docker + Kubernetes | 部署运维 |

### 3.3 AI技术栈

| 领域 | 技术选型 | 说明 |
|------|---------|------|
| **大语言模型** | GPT-4o / 本地部署Qwen/DeepSeek | 辨证论治、方剂推荐 |
| **视觉模型** | ResNet / ViT (微调) | 舌诊、面色分析 |
| **语音分析** | Wav2Vec / HuBERT | 声诊特征提取 |
| **知识图谱** | Neo4j / NebulaGraph | 中医辨证推理链路 |
| **向量数据库** | Milvus / PGVector | 相似病例检索 |
| **模型管理** | MLflow / BentoML | 模型版本管理 |

### 3.4 数据存储

| 类型 | 方案 | 用途 |
|------|------|------|
| **关系型数据库** | PostgreSQL | 患者信息、就诊记录、处方 |
| **文档数据库** | MongoDB | 四诊原始数据（JSON格式） |
| **对象存储** | MinIO / S3 | 舌诊图片、脉波图、病历附件 |
| **缓存** | Redis | Token、临时数据、设备状态 |
| **时序数据库** | InfluxDB | 脉诊波形等时序指标 |
| **向量数据库** | PGVector (PostgreSQL插件) | 病例向量、症状向量 |
| **搜索引擎** | Elasticsearch | 病历检索、知识库搜索 |

## 4. 核心模块设计

### 4.1 四诊采集服务

```
设备连接层 ──▶ 数据预处理 ──▶ 特征提取 ──▶ 融合编码 ──▶ 上行云端
     │            │              │              │
  ┌──┴──┐    ┌────┴────┐   ┌───┴────┐   ┌────┴────┐
  │USB  │    │降噪 去抖  │   │舌体分割 │   │四诊特征  │
  │蓝牙 │    │归一化    │   │脉象参数 │   │向量拼接  │
  │WiFi │    │数据校验  │   │声纹特征 │   │JSON编码  │
  └─────┘    └─────────┘   └────────┘   └─────────┘
```

**设备接入规范：**
- 定义统一的 `DiagnosisDevice` 抽象接口
- 各设备厂商实现该接口的SDK适配器
- 支持热插拔检测、设备校准状态管理

### 4.2 AI诊断服务（核心）

```
四诊数据 ──▶ 特征融合 ──▶ LLM推理 ──▶ 结果精炼 ──▶ 诊断报告
                              │
                ┌─────────────┼─────────────┐
                ▼             ▼             ▼
            证型判断       治则治法       方剂推荐
            (Prompt+       (Prompt+       (RAG+
             结构化输出)     知识图谱)      向量检索)
```

**LLM推理策略：**

| 阶段 | 方法 | 说明 |
|------|------|------|
| **Prompt Engineering** | 结构化System Prompt + Few-shot | 将四诊参数映射为中医辨证特征描述 |
| **RAG增强** | 知识图谱 + 向量检索 | 检索相似病例、经典方剂作为参考 |
| **结构化输出** | JSON Schema约束 | 确保输出格式可被下游系统解析 |
| **合议机制** | Multi-model voting | 多个模型输出进行加权合议（可选） |

**Prompt设计思路（简化示例）：**
```
你是一位资深中医专家，请基于以下四诊信息进行辨证论治：

望诊：{舌质：淡红，舌苔：薄白，面色：...}
闻诊：{声低气怯，...}
问诊：{畏寒肢冷，纳差便溏，...}
切诊：{脉象：沉细，脉率：68次/分，...}

请以JSON格式输出：
{
  "zhengxing": "证型名称及依据",
  "zhize": "治则治法",
  "fangji": {"name": "方剂名称", "composition": [...], "dosage": {...}},
  "acupuncture": {...},
  "health_advice": "..."
}
```

### 4.3 设备管理服务

| 模块 | 功能 |
|------|------|
| **设备注册** | 设备厂商、型号、序列号登记 |
| **设备校准** | 校准状态管理、定期校准提醒 |
| **状态监控** | 在线/离线、连接质量、数据质量评分 |
| **驱动管理** | 驱动版本管理、自动更新 |
| **日志记录** | 设备操作日志、异常日志 |

## 5. 安全架构

### 5.1 数据安全分层

```
┌──────────────────────────────┐
│        应用层安全               │
│  JWT认证 + RBAC权限 + 审计日志  │
├──────────────────────────────┤
│        传输层安全               │
│  TLS 1.3 + mTLS (服务间)      │
├──────────────────────────────┤
│        存储层安全               │
│  AES-256加密 + HSM密钥管理     │
├──────────────────────────────┤
│        数据脱敏                 │
│  患者姓名/身份证/手机号动态脱敏  │
└──────────────────────────────┘
```

### 5.2 合规要点
- 诊断结论必须经医生确认签署，AI仅作为辅助
- 所有AI生成内容标注"AI辅助建议，仅供参考"
- 患者数据存储遵循《个人信息保护法》
- 系统部署支持三级等保要求

## 8. 风险与对策

| 风险 | 影响 | 概率 | 对策 |
|------|------|------|------|
| 设备厂商SDK不兼容 | 采集端开发延期 | 高 | 提前与厂商沟通，预留抽象层 |
| LLM输出幻觉 | 诊断准确率低 | 中 | 知识图谱约束 + 医生确认机制 |
| 患者数据安全合规 | 法律风险 | 中 | 等保三级方案 + 数据脱敏 |
| 医生接受度低 | 推广困难 | 中 | UI设计符合医师习惯 + 培训支持 |
| 网络不稳定 | 离线体验差 | 低 | 完善的离线/同步机制 |
